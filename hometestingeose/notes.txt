SECTION 1: OPENPLC v4
=============================================================

What it is:
  OpenPLC v4 is a headless industrial PLC runtime. It runs IEC 61131-3
  programs (Ladder Logic, Structured Text, Function Block Diagram) on
  standard Linux hardware. It has two components: a Python/Flask REST API
  server on port 8443, and a C++ real-time execution engine.

SECTION 3: ASYNC RUST AND TOKIO
=============================================================

Why async exists in Rust:
  Rust does not ship a built-in async runtime. The language provides
  async/await syntax but something has to actually run the futures.
  Tokio is that runtime. It provides a thread pool, an event loop,
  timers, async file I/O, async network I/O, and task spawning.
  Without Tokio, async fn compiles but does nothing.

Tokio official tutorial (READ THIS — it is the best resource for
understanding async Rust in a practical context. Work through it
chapter by chapter the same way you did the Rust Book):
  https://tokio.rs/tokio/tutorial

  Most relevant chapters for your project:
    Hello Tokio       — you already did this
    Spawning          — tokio::spawn, concurrent tasks (needed for main.rs)
    Shared state      — how tasks share data (relevant for watch channel)
    Channels          — mpsc, oneshot, watch (needed for heartbeat later)
    Select            — waiting for multiple futures (needed for shutdown)
    Graceful shutdown — how to handle SIGTERM cleanly (needed for Docker)

Tokio crate documentation:
  https://docs.rs/tokio/latest/tokio/

Why tokio::time::sleep instead of std::thread::sleep:
  std::thread::sleep blocks the OS thread entirely — no other async
  task can run on that thread while it sleeps.
  tokio::time::sleep yields the thread back to Tokio's scheduler so
  other tasks can run. Always use the Tokio version in async code.

Why tokio::fs::read instead of std::fs::read:
  Same reason — std::fs::read blocks the thread doing disk I/O.
  tokio::fs::read yields while waiting for the disk. Use Tokio's
  version in async code.

SECTION 5: NETWORKING CONCEPTS
=============================================================

HTTP and REST APIs:
  Good foundational resource:
    https://developer.mozilla.org/en-US/docs/Web/HTTP/Overview
  
  Key concepts you need:
    - HTTP methods: GET, POST (you use both)
    - Status codes: 200 OK, 201 Created, 400 Bad Request,
      401 Unauthorized, 404 Not Found, 409 Conflict,
      500 Internal Server Error
    - Headers: Content-Type, Authorization (Bearer tokens)
    - Request body formats: JSON, multipart/form-data (file uploads)
    - HTTPS vs HTTP: TLS encryption, certificates, why self-signed
      certs need special handling

JWT (JSON Web Tokens):
  What they are and how they work:
    https://jwt.io/introduction
  
  In your project: OpenPLC returns a JWT after login. You attach it
  to every subsequent request as "Authorization: Bearer <token>".
  The token encodes your identity and has an expiry (24 hours by
  default in OpenPLC). After expiry you need to login again.
  Your current code does not handle token expiry — something to
  add later.

TLS and self-signed certificates:
  OpenPLC generates a self-signed certificate on first run stored at
  webserver/certOPENPLC.pem. Self-signed means it was not issued by
  a trusted Certificate Authority. Your reqwest client is configured
  with danger_accept_invalid_certs(true) to bypass verification.
  
  In production you would instead extract the cert from the container
  and add it to your trust store, then use:
    .add_root_certificate(cert)
  instead of bypassing verification entirely.

Modbus TCP:
  What it is: an industrial protocol for reading/writing PLC registers
  over TCP/IP. Runs on port 502. Your future external program will
  connect to each container's exposed Modbus port to read sensor data.
  
  Protocol overview:
    https://modbus.org/docs/Modbus_Application_Protocol_V1_1b3.pdf
  
  Key concepts:
    - Coils (single-bit read/write)
    - Discrete inputs (single-bit read-only)
    - Holding registers (16-bit read/write)
    - Input registers (16-bit read-only)
    - Function codes (01, 02, 03, 04, 05, 06, 15, 16)
    - Unit ID / slave address
  
  OpenPLC maps its IEC 61131-3 variables to Modbus registers
  automatically based on variable addresses (%IX, %QX, %IW, %QW).
  
  Rust crate for Modbus TCP client (you will need this later):
    tokio-modbus: https://docs.rs/tokio-modbus/latest/tokio_modbus/
    GitHub: https://github.com/slowtec/tokio-modbus

SECTION 6: DOCKER
=============================================================

Official Docker documentation (start here):
  https://docs.docker.com/get-started/

  Most relevant sections for your project:
    Overview:         https://docs.docker.com/get-started/overview/
    Building images:  https://docs.docker.com/build/guide/
    Dockerfile ref:   https://docs.docker.com/reference/dockerfile/
    Multi-stage:      https://docs.docker.com/build/building/multi-stage/
    Volumes:          https://docs.docker.com/storage/volumes/
    Networking:       https://docs.docker.com/network/
    Docker Compose:   https://docs.docker.com/compose/

Key concepts for your project:

  Multi-stage builds:
    You use two stages. Stage 1 uses rust:latest to compile your binary.
    Stage 2 uses ubuntu:22.04 and copies only the compiled binary from
    stage 1. The compiler (1.5GB) never makes it into the final image.
    This is essential — without it your image would be enormous.

  Entrypoint vs CMD:
    ENTRYPOINT is the process that runs when the container starts.
    CMD provides default arguments to ENTRYPOINT.
    You use a shell script as ENTRYPOINT that starts OpenPLC in the
    background then exec's your Rust binary as PID 1.

  PID 1 and signal handling:
    Docker sends SIGTERM to PID 1 when you run docker stop.
    If your Rust binary is PID 1 it receives SIGTERM directly and
    can shut down gracefully. If a shell script is PID 1 and your
    Rust binary is a child process, the signal may not be forwarded
    correctly. The exec command in entrypoint.sh replaces the shell
    with your binary, making it PID 1.

  Volumes:
    OpenPLC stores its database and compiled programs in
    /var/run/runtime/ inside the container. Without a volume this
    data is lost every time the container restarts — meaning OpenPLC
    would have no users and no compiled program on every restart.
    Mounting a named volume at /var/run/runtime/ persists this data
    across restarts.

  Port mapping:
    -p host_port:container_port
    Inside the container everything uses container ports (8443, 502).
    Your Rust binary always connects to localhost:8443 regardless of
    what host port is mapped. Port mapping only affects external access.

  Environment variables in Docker:
    Three ways to set them (in order of priority, highest first):
      1. docker run -e KEY=value (runtime, highest priority)
      2. docker-compose.yml environment section
      3. Dockerfile ENV instruction (baked-in defaults, lowest priority)

  Capabilities:
    --cap-add=SYS_NICE and --cap-add=SYS_RESOURCE are required for
    OpenPLC to use SCHED_FIFO real-time scheduling. Without them
    OpenPLC still runs but without deterministic timing guarantees.

  Docker Compose for multiple containers:
    When you scale to N containers you will use docker-compose.yml
    to define all of them. Each service is an identical image with
    different environment variables and port mappings. This is the
    endgame for your multi-node setup.

SECTION 7: DESIGN DECISIONS (no external resource exists)
=============================================================

These are decisions made in your codebase and why.

--- Why one container per PLC node instead of one container total ---
  Each container is completely isolated — its own OpenPLC process,
  its own database, its own compiled program, its own Modbus slave.
  Isolation means one node crashing does not affect others. It also
  means each node can run a different PLC program in the future just
  by building a different image. Scaling is just adding more containers.

--- Why the program.zip is baked into the image ---
  Baking it in means each image version is a complete self-contained
  artifact. You can tag it (plc-node:v1.2.3), roll back by running
  an older image, and be certain production exactly matches what you
  tested. The alternative (mounting the zip as a volume) introduces
  the possibility of mismatch between the binary and the program.

--- Why Rust for the controller instead of Python ---
  The Python script shown earlier works fine. Rust gives you:
    - A single compiled binary with no runtime dependency (no Python
      interpreter needed in the container)
    - Smaller container image
    - Memory safety guarantees
    - Native async via Tokio (same ecosystem as your MJDJS project)
    - Good practice for your internship preparation
  The tradeoff is more complexity upfront, which is why you are
  learning it carefully.

--- Why reqwest with rustls instead of OpenSSL ---
  rustls is a pure-Rust TLS implementation. It requires no system
  library (no libssl-dev in the container). OpenSSL requires system
  packages, introduces version compatibility issues, and adds attack
  surface. For a container that communicates only with localhost,
  rustls is simpler and sufficient.

--- Why danger_accept_invalid_certs instead of importing the cert ---
  OpenPLC regenerates its self-signed cert on first run. The cert
  changes between fresh container deployments. Importing it would
  require extracting it after first run and rebuilding the image or
  mounting it as a volume. For a system where both processes are in
  the same container and communication never leaves localhost, the
  security tradeoff of skipping cert verification is acceptable.
  In a setup where the Rust binary and OpenPLC were in separate
  containers communicating over a real network, you would extract
  and import the cert properly.

--- Why create_user() treats 409 as success ---
  The container may restart against an existing volume where the user
  already exists. If 409 were treated as an error, every restart after
  the first would fail immediately. Treating it as success makes the
  deploy sequence idempotent — safe to run multiple times with the
  same result.

--- Why auth_header() panics instead of returning Result ---
  auth_header() is only called from authenticated methods. Authenticated
  methods are only called after login(). If auth_header() is called
  before login() that is a bug in your code, not a runtime condition.
  Returning Result would force every calling method to handle an error
  that should be impossible in correct code. A panic with a clear
  message is the right response to a programming error.

--- Why Config reads all env vars at startup instead of on demand ---
  Reading at startup gives you one clear moment where misconfiguration
  is caught — immediately when the container starts, before any HTTP
  calls are made. If PLC_PASSWORD is missing you find out in the first
  second, not halfway through a deploy sequence. It also means the rest
  of your code never has to deal with missing config — it is guaranteed
  to be present if the program is running.

--- Why private fields on PlcClient ---
  jwt_token and auth_password are sensitive. Making them private means
  no code outside client.rs can read or accidentally overwrite them.
  Outside code interacts with PlcClient only through its public methods.
  This is encapsulation — the internal state is an implementation detail.

--- Future: token expiry handling ---
  JWT tokens expire after 24 hours by default in OpenPLC. Your current
  code logs in once and uses that token forever. For a long-running
  container this will eventually fail. Future improvement: check the
  token expiry before each request and re-login if it is close to
  expiring. The token is a JWT so you can decode its exp claim without
  making a network call.

--- Future: Modbus client in Rust ---
  When you add Modbus reading, use the tokio-modbus crate. It is async
  and integrates naturally with your existing Tokio setup. You will
  add a new file modbus.rs with a ModbusClient struct that connects to
  localhost:502, reads registers on an interval, and forwards data to
  your external system. This becomes a third concurrent task in main.rs
  alongside deploy and heartbeat.

--- Future: multiple containers with docker-compose ---
  Each container is an identical image. docker-compose.yml defines N
  services, each with a unique NODE_ID and unique host port mappings.
  The external system identifies each node by NODE_ID in the heartbeat
  payload and by host port for Modbus connections. The Rust code and
  image are identical across all nodes.

--- Future: health checks ---
  Add a HEALTHCHECK instruction to the Dockerfile:
    HEALTHCHECK CMD curl -k https://localhost:8443/api/ping || exit 1
  Docker will periodically run this and mark the container unhealthy
  if it fails. Useful for monitoring and for docker-compose depends_on
  conditions.
